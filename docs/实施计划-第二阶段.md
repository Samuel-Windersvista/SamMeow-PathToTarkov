## 阶段 1：紧急修复（用户可见故障）

> 目标：修复玩家直接感知的运行时故障。这些故障会导致撤离不可用、存档损坏、配置校验盲区或战后流程异常。按严重程度排序，但实施顺序可并行。

### 1.1 CRITICAL-1：CurrentLocationDataService 静默失败导致全部撤离点禁用

- **文件**：`PTT-Plugin/Services/CurrentLocationDataService.cs`、`PTT-Plugin/Plugin.cs`
- **风险**：LOW
- **共识度**：95%
- **估算改动**：~15 行

**根因分析**：

`Init()` 方法在第 14-25 行的控制流存在致命缺陷：`FetchExfilsTargetsForCurrentLocation()` 内部 try/catch 吞掉了 HTTP 异常，调用方无法获悉初始化是否成功，`_isInitialized` 被无条件置为 `true`。当服务器首次启动时后端未就绪或网络不通，`CurrentLocationData` 保持为空字典，`IsExfiltrationPointEnabled()` 对全部撤离点返回 `false`。玩家进入战局后所有撤离点灰显，无法撤离。

此外，`Reset()` 方法虽然能将 `_isInitialized` 清零，但仅在 `KeepFoundInRaidTweak` 的 checkmark 补丁路径中被调用，没有自动重试机制。这意味着一旦首次初始化失败，故障将持续到进程重启或触发到 Reset 路径。

**实施细节**：

1. 将 `FetchExfilsTargetsForCurrentLocation()` 的返回类型从 `void` 改为 `bool`，成功返回 `true`，失败返回 `false`。
2. 修改 `Init()` 签名从 `void Init()` 改为 `bool Init()`，仅在 fetch 成功时设置 `_isInitialized = true` 并调用 `TryApplyExfilFiltering()`。
3. 在 `Plugin.cs` 中调用 `Init()` 的位置检查返回值，失败时输出 Error 日志但**不阻塞**服务启动——保留原版撤离行为作为降级策略。
4. 不引入指数退避重试（经三议员讨论决定：首次修复保持最小改动，避免引入定时器相关的生命周期问题。重试能力留给后续迭代）。

```csharp
// CurrentLocationDataService.cs
public bool Init()
{
    if (_isInitialized) return true;
    bool success = FetchExfilsTargetsForCurrentLocation();
    if (!success)
    {
        Logger.Error("[PTT] CurrentLocationDataService init failed, exfil filtering disabled");
        return false;
    }
    _isInitialized = true;
    PTT.Patches.InitAllExfiltrationPointsPatch.TryApplyExfilFiltering();
    return true;
}
```

```csharp
// Plugin.cs 调用处
bool initOk = currentLocationDataService.Init();
if (!initOk)
{
    Logger.Warning("[PTT] Running with vanilla exfil behavior (PTT filtering disabled)");
}
```

**回退方式**：还原 `Init()` 和调用处的改动，单文件可逆。

### 1.2 HIGH-1：双阶段 saveProfile 导致存档不一致

- **文件**：`src/keep-fir-tweak.ts`、`src/path-to-tarkov-controller.ts`
- **风险**：LOW
- **共识度**：93%
- **估算改动**：~12 行

**根因分析**：

`onPlayerExtracts()` 的执行路径中存在两次 `saveProfile()` 调用：第一次在 `KeepFoundInRaidTweak.setFoundInRaidOnEquipment()` 内部（标记 FIR 后立即保存），第二次在 `updateOffraidPosition()` 末尾（更新 stash 和 trader 后保存）。两次保存之间是一个脆弱的窗口：如果第一次保存后、第二次保存前服务器崩溃或网络中断，存档处于半更新状态。物品的 FIR 标记已写入但 stash 切换和 trader 状态未更新，玩家重新进入后手持已标记 FIR 的物品但仓库位置回退到战前状态，可能导致物品逻辑复制。

**实施细节**：

1. 从 `KeepFoundInRaidTweak.setFoundInRaidOnEquipment()` 中移除 `saveServer.saveProfile()` 调用。该方法只负责修改内存中的 profile 数据，不负责持久化。
2. 在 `PathToTarkovController.onPlayerExtracts()` 中，确认 `updateOffraidPosition()` 是唯一的 `saveProfile()` 调用点。
3. 添加注释标注：`// saveProfile 由 updateOffraidPosition 统一管理，请勿在 FIR tweak 中单独保存`。

```typescript
// keep-fir-tweak.ts — 移除 save
public setFoundInRaidOnEquipment(sessionId: string, isPlayerScav: boolean): number {
    const profile = this.saveServer.getProfile(sessionId);
    // ... 修改物品的 FIR 标记 ...
    // [移除] this.saveServer.saveProfile(sessionId);
    return nbImpactedItems;
}

// path-to-tarkov-controller.ts — 确认单一 save 路径
onPlayerExtracts(sessionId, isPlayerScav, newOffraidPosition): void {
    if (this.userConfig.gameplay.keepFoundInRaidTweak) {
        const firTweak = new KeepFoundInRaidTweak(this.saveServer);
        firTweak.setFoundInRaidOnEquipment(sessionId, isPlayerScav);
        // saveProfile 由 updateOffraidPosition 统一管理
    }
    this.updateOffraidPosition(sessionId, newOffraidPosition);
}
```

**回退方式**：恢复 `setFoundInRaidOnEquipment` 中被移除的 `saveProfile` 调用。

### 1.4 HIGH-5：战后 transit/extract 判定逻辑缺陷

- **文件**：`src/event-watcher.ts`、`src/end-of-raid-controller.ts`
- **风险**：LOW-MEDIUM
- **共识度**：85%
- **估算改动**：~25 行

**根因分析**：

该问题包含三个相互关联的子缺陷：

**子缺陷 A**（`event-watcher.ts` 第 297 行）：`getEndOfRaidPayload()` 中 `isTransit` 字段完全依赖 `!targetOffraidPosition` 进行推导，而非使用 `exitStatus === 'Transit'`。在 vanilla transit 场景下，如果 exitName 不包含 PTT 格式的 `.` 分隔信息，`targetOffraidPosition` 可能为 null，导致 `isTransit` 被错误设为 `true`。

**子缺陷 B**（`event-watcher.ts` 第 300 行调用条件）：`handleRegularExtracts()` 的调用条件是 `exitName && !targetOffraidPosition && !transitTargetMapName && !transitTargetSpawnPointId`，这个条件在 vanilla transit 场景下也会被满足——一个有 exitName 但不是 PTT 格式的出口会被错误路由到 `handleRegularExtracts` 回退逻辑。

**子缺陷 C**（`end-of-raid-controller.ts` 第 54 行）：`newOffraidPosition` 非空时调用 `onPlayerExtracts` 触发 FIR tweak 和位置更新，但 `isTransit` 为 true 时只记日志不执行任何动作。返回的玩家得不到 offraidPosition 更新。

**实施细节**：

1. 在 `getEndOfRaidPayload()` 中将 `isTransit` 判定改为优先使用 `raidCache.exitStatus`：
   ```typescript
   const isTransit = raidCache.exitStatus === 'Transit'
       || (!targetOffraidPosition && transitTargetMapName !== null);
   ```

2. 在 `handleRegularExtracts()` 调用条件中增加守卫：
   ```typescript
   if (exitName && !targetOffraidPosition && !transitTargetMapName && !transitTargetSpawnPointId
       && raidCache.exitStatus !== 'Transit') {
       return this.handleRegularExtracts(endOfRaidPayload);
   }
   ```

3. 在 `end-of-raid-controller.ts` 中为 transit 路径补充 offraidPosition 保持逻辑：当 `isTransit` 为 true 且 `newOffraidPosition` 为 null 时，保持当前 `offraidPosition` 不变写入存档，避免位置丢失。

**回退方式**：三项改动分别在三个函数中，可单独还原。建议如需回滚，子缺陷 A+B 一起恢复（共用 transit 判定流），子缺陷 C 可单独恢复。

## 阶段 2：正确性修复（错误行为）

> 目标：修复不会直接导致崩溃但造成逻辑错误的问题。这些问题在特定配置或运行时条件下触发，影响数据一致性和功能正确性。

### 2.1 HIGH-2：CurrentExfilTargetService 全局静态单例污染

- **文件**：`PTT-Plugin/Services/CurrentExfilTargetService.cs`、`PTT-Plugin/Services/CustomExfilService.cs`
- **风险**：LOW
- **共识度**：92%
- **估算改动**：~20 行

**根因分析**：

`CurrentExfilTargetService` 被设计为 `public static class`，其中的 `CurrentExfilTarget` 静态属性是整个进程级别的全局状态。在 SPT 的 cooperative 模式或 headless 客户端场景中，单个服务器进程服务多个玩家会话。静态变量会在不同玩家的撤离请求之间共享：

- 玩家 A 调用 `SaveExfil(targetA)`，写入静态字段。
- 玩家 B 随后调用 `SaveExfil(targetB)`，覆盖静态字段。
- 玩家 A 的撤离请求如果因网络延迟稍后才消费，`ConsumeExitName()` 返回的可能是玩家 B 的数据。
- `SaveExfil` 在确认 TransitController 存在之前就执行了保存操作，失败路径没有任何清理逻辑。

**实施细节**：

1. 将 `CurrentExfilTarget` 的存储从单字段改为 `ConcurrentDictionary<string, ExfilTarget>`，以 `sessionId` 为键。
2. `SaveExfil(sessionId, target)` 仅在确认 TransitController 响应成功后才写入字典。
3. 新增 `CleanupSession(sessionId)` 方法，在玩家会话结束时清除对应条目。
4. 现有 `Reset()` 方法改为遍历字典清除所有条目（保留兼容性）。

```csharp
public static class CurrentExfilTargetService
{
    private static readonly ConcurrentDictionary<string, ExfilTarget> _targets = new();

    public static void SaveExfil(string sessionId, ExfilTarget target)
    {
        // 验证通过后才写入
        _targets[sessionId] = target;
    }

    public static string? ConsumeExitName(string sessionId)
    {
        if (_targets.TryRemove(sessionId, out var target))
            return target.ExitName;
        return null;
    }

    public static void CleanupSession(string sessionId)
    {
        _targets.TryRemove(sessionId, out _);
    }

    public static void Reset()
    {
        _targets.Clear();
    }
}
```

**回退方式**：还原为静态单字段模式，单文件可逆。

### 2.2 MEDIUM-1：checkAccessVia 子串匹配 bug

- **文件**：`src/helpers.ts`
- **风险**：LOW
- **共识度**：97%
- **估算改动**：~5 行

**根因分析**：

`checkAccessVia()` 函数的第二个参数 `value` 来源于配置中的 `access_via` 字段。该字段的类型是 `string | string[]`。当值为字符串时，JavaScript 的 `String.prototype.includes()` 执行的是子串匹配而非精确值匹配：

- 示例：`access_via = "factory4_day"`，`value = "factory"` => `"factory4_day".includes("factory")` 返回 `true`，但语义上 `"factory"` 并不是一个有效的 access_via 值。
- 在配置中，如果某个 offraidPosition 名称恰好是另一个 access_via 值的子串（如 `"MY_TOWN"` 和 `"MY_TOWNHALL"`），就会产生误匹配。

此外，`isWildcardAccessVia()` 对单字符的通配符判断也有问题：当 `access_via` 是字符串 `"*"` 时，`access_via.length === 1 && access_via[0] === '*'` 虽然能正确识别，但如果 `access_via` 是一个包含 `"*"` 的数组，行为不一致。

**实施细节**：

```typescript
export function checkAccessVia(access_via: AccessVia, value: string): boolean {
    const viaArray = Array.isArray(access_via) ? access_via : [access_via];
    return isWildcardAccessVia(viaArray) || viaArray.includes(value);
}

export function isWildcardAccessVia(access_via: AccessVia): boolean {
    const arr = Array.isArray(access_via) ? access_via : [access_via];
    return arr.length === 1 && arr[0] === '*';
}
```

改动说明：`checkAccessVia` 将输入统一转为数组后进行精确 `includes()` 匹配，消除字符串和数组类型之间的语义差异。`isWildcardAccessVia` 同样统一处理类型，确保 `*` 通配符行为一致。

**回退方式**：还原 `helpers.ts` 中的两个函数定义。

### 2.3 MEDIUM-5：createGetTemplateItems 模板缺失硬崩溃

- **文件**：`src/path-to-tarkov-controller.ts`
- **风险**：LOW
- **共识度**：96%
- **估算改动**：~8 行

**根因分析**：

`createGetTemplateItems()` 方法遍历 `VANILLA_STASH_IDS`，逐项访问 `items[stashId]._props.Grids[0]._props`。它假设所有 ID 在数据库的 `templates.items` 中一定存在且结构完整。当 SPT 升级后移除了某个 stash template，或修改了 items 的数据结构时，`item?._props?.Grids?.[0]` 链中的任意一环为 `undefined` 都会导致 `throw new Error(...)`。由于该方法在 `DataCallbacks.getTemplateItems` 的 override 路径中被调用，任何玩家打开仓库都会触发这个路由，一旦抛出异常，该玩家的全部 HTTP 请求都会得到 500 错误，表现为仓库完全无法打开。

**实施细节**：

将 `throw new Error(...)` 替换为 `debug()` 日志 + `return` 优雅跳过：

```typescript
VANILLA_STASH_IDS.forEach(stashId => {
    const item = items[stashId];
    if (!item) {
        this.debug(`stash template ${stashId} not found, skipping`);
        return;
    }
    const grid = item._props?.Grids?.[0];
    const gridProps = grid?._props;
    if (gridProps) {
        gridProps.cellsV = size;
    } else {
        this.debug(`cannot set size for stash ${stashId}, missing grid props`);
    }
});
```

**降级效果**：某个 stash template 缺失或结构异常时，该 stash 保持原版大小不变而非崩溃。玩家仍然可以正常打开仓库和进行操作。排查者可在 debug 日志中看到具体的 stash ID，便于定位数据库问题。

**回退方式**：恢复为 `throw new Error` 方式。

## 阶段 3：架构加固（长期健康）

> 目标：改善代码架构的可持续性和可维护性。这些问题在短期内不会引发用户可见故障，但在长期运行、多玩家并发或版本升级场景下会积累技术债务。

### 3.1 HIGH-3：headless 客户端用户名匹配串位

- **文件**：`src/path-to-tarkov-controller.ts` — `getOffraidPosition()` 方法
- **风险**：MEDIUM
- **共识度**：78%
- **估算改动**：~30 行

**根因分析**：

`getOffraidPosition()` 在第 1073-1103 行通过 `profile.info.username.includes('headless')` 来识别 headless 客户端。找到 headless profile 后，遍历 `saveServer.getProfiles()` 中所有 profiles，返回第一个非 headless 且有 `offraidPosition` 的 profile 的数据。这个方案有三个问题：

1. **误判风险**：任何用户名包含 "headless" 子串的正常玩家都会被误判为 headless 客户端。
2. **匹配不确定性**：在多玩家环境下遍历全部 profiles 取第一个匹配，如果存在多个非 headless 玩家，可能返回错误的 offraidPosition。
3. **性能开销**：每次调用 `getOffraidPosition()` 都扫描全部 profiles，在多人会话多的服务器上产生 O(n) 扫描。

**实施细节**：

1. 在 `PathToTarkovController` 中新增 `Map<string, string>` 类型的字段 `headlessSessionMap`，映射关系为 `headlessSessionId -> hostSessionId`。
2. 在 raid 创建时（通过 `createInitialRaidCache` 或类似入口）注册 headless 与主机玩家的 session 配对关系。配对的来源可以是 PTT-Fika 模块在客户端连接时传递的显式映射。
3. `getOffraidPosition()` 中优先查询 `headlessSessionMap` 获取主机 sessionId，然后直接读取主机 profile 的 offraidPosition。
4. 保留 `username.includes('headless')` 作为**降级 fallback**，但不再 mutate headless profile 的数据，仅作为只读探测。

```typescript
private headlessSessionMap: Map<string, string> = new Map();

// raid 创建时注册（在 PTT-Fika 配合下调用）
public registerHeadlessPair(headlessSessionId: string, hostSessionId: string): void {
    this.headlessSessionMap.set(headlessSessionId, hostSessionId);
}

public getOffraidPosition(sessionId: string): OffraidPosition | undefined {
    // 优先查映射表
    const hostSessionId = this.headlessSessionMap.get(sessionId);
    if (hostSessionId) {
        const hostProfile = this.saveServer.getProfile(hostSessionId);
        return hostProfile?.PathToTarkov?.offraidPosition;
    }

    // fallback：用户名匹配（不移除，兼容无 PTT-Fika 场景）
    const profile = this.saveServer.getProfile(sessionId);
    const profileName = profile?.info?.username || '';
    if (profileName.toLowerCase().includes('headless')) {
        // ... 现有 fallback 逻辑，但只读不写 ...
    }
    return profile?.PathToTarkov?.offraidPosition;
}
```

**依赖关系**：需要 PTT-Fika 模块配合在适当生命周期调用 `registerHeadlessPair()`。如果 PTT-Fika 尚未修改，仅 fallback 路径生效，行为与当前一致（无退化）。

**争议说明**：议会中 22% 的反对票认为该改动引入了模块间耦合（PTT-Fika 需要知道 `headlessSessionMap` 的存在），且 fallback 路径保留增加了维护成本。共识决定：接受耦合，因为这是根治 headless 匹配问题的必经之路，fallback 作为过渡方案将在下个版本中移除。

**回退方式**：移除 `headlessSessionMap` 相关代码，恢复为纯用户名匹配。由于新增字段不影响现有逻辑，回退时只需删除新增代码。

### 3.2 MEDIUM-2：raidCaches/configCache 无清理导致内存泄漏

- **文件**：`src/event-watcher.ts`、`src/path-to-tarkov-controller.ts`
- **风险**：LOW
- **共识度**：94%
- **估算改动**：~15 行

**根因分析**：

`EventWatcher` 中的 `raidCaches: Record<string, RaidCache>` 和 `PathToTarkovController` 中的 `configCache: Record<string, Config>` 是两个用 sessionId 索引的缓存字典。在 SPT 的长期运行场景中（服务器持续运行数小时，服务数百个不同玩家会话），这两个字典只增不减。每个 `RaidCache` 对象包含 `currentLocationName`、`exitStatus`、`spawnPoint` 等字段；每个 `Config` 对象是深度克隆后的完整配置树。随着时间推移，内存占用线性增长，可能触发 GC 压力或 OOM。

`raidCaches` 的问题更突出：每次玩家开局都在 `initRaidCache` 中创建新条目（transit 场景下跳过了 reset，但不会清理旧条目），玩家断开后条目永久残留。

**实施细节**：

1. **configCache 剪枝**：在 `getConfig()` 写入新缓存时检查 `Object.keys(configCache).length > 100`，触发时将条目数裁剪到 50。裁剪采用 LRU 近似策略——保留最近访问的 50 个条目（简单的做法：收集所有 key，按访问顺序排序后删除后半部分；更精确的做法见下方伪代码）。
2. **raidCache 自然替换**：`raidCaches` 改用 `Map`，在 `initRaidCache` 中当 `exitStatus !== 'Transit'` 时直接用 `set` 覆盖同 sessionId 的旧条目。非 transit 开局时旧条目被自然替换。
3. **清理钩子**：在 `EventWatcher` 中新增 `cleanupSession(sessionId)` 方法，由外部在会话结束时调用，从两个缓存中移除对应条目。

```typescript
// configCache 剪枝
private getConfig(sessionId: string): Config {
    // ... 现有逻辑 ...
    // 写入缓存后检查大小
    const keys = Object.keys(this.configCache);
    if (keys.length > 100) {
        const toDelete = keys.slice(0, keys.length - 50); // 删掉旧的一半
        for (const key of toDelete) {
            delete this.configCache[key];
        }
        this.logger.debug(`[PTT] configCache pruned: ${keys.length} -> ${Object.keys(this.configCache).length}`);
    }
    return this.configCache[sessionId];
}

// EventWatcher 清理钩子
public cleanupSession(sessionId: string): void {
    delete this.raidCaches[sessionId];
    // 通知 controller 清理 configCache（通过回调或直接调用）
}
```

**回退方式**：移除剪枝逻辑和清理钩子调用，恢复为无界字典。

### 3.3 MEDIUM-6：构造函数覆盖注入依赖

- **文件**：`src/path-to-tarkov-controller.ts`
- **风险**：ZERO
- **共识度**：100%
- **估算改动**：2 行

**根因分析**：

构造函数第 89 行接受 `tradersAvailabilityService: TradersAvailabilityService` 作为参数注入，但在第 112 行立即用 `new TradersAvailabilityService()` 创建新实例覆盖了该参数。这个 bug 意味着：

**实施细节**：

删除第 112 行覆盖赋值：

```typescript
// 删除以下行：
// this.tradersAvailabilityService = new TradersAvailabilityService();

// 保留构造函数参数中注入的实例
constructor(
    // ...
    private tradersAvailabilityService: TradersAvailabilityService,
    // ...
)
```

**注意事项**：需要确认调用方传入的 `tradersAvailabilityService` 确实是已正确初始化的实例。经代码审查确认，`mod.ts` 或 `modLoader.ts` 中创建 `PathToTarkovController` 时传入的实例是单例且已初始化，删除内部 `new` 后行为等价。

**回退方式**：恢复 `this.tradersAvailabilityService = new TradersAvailabilityService();` 行。

### 3.4 MEDIUM-7：Debug 日志在 info 级别污染

- **文件**：`src/path-to-tarkov-controller.ts`（14 处）、`src/event-watcher.ts`（5 处）、及其他文件中零散的 `[PTT Debug]` 日志
- **风险**：ZERO
- **共识度**：91%
- **估算改动**：~20 行

**根因分析**：

代码库中大量以 `[PTT Debug]` 为前缀的日志使用 `this.logger.info()` 输出，包括每次 raid 同步前后的 `SpawnPointParams` 长度、player spawn 点完整列表、`syncLocationBase` 调用详情等。在正常运行的服务器中，每次 raid 开始会输出 10-20 条此类日志。问题在于：

- 使用 `info` 级别意味着服务器管理员无法通过常规的日志级别配置过滤（除非将全局日志级别设为 `warn` 或更高，但这会丢失真正的 info 级信息）。
- 日志字符串即使在不输出时也会被构造（JavaScript 模板字符串的求值），产生不必要的字符串拼接开销。
- 对最终用户没有诊断价值（debug 信息只对模组开发者有用）。

**实施细节**：

全局搜索 `[PTT Debug]` 模式，将所有使用 `this.logger.info()` 或 `this.logger.warning()` 输出 debug 信息的调用替换为 `this.logger.debug()`。具体修改模式：

```typescript
// 改动前
this.logger.info(`[PTT Debug] syncLocationBase called for sessionId: ${sessionId}`);

// 改动后
this.logger.debug(`[PTT Debug] syncLocationBase called for sessionId: ${sessionId}`);
```

**批量替换范围**：

| 文件 | 替换处数 | 说明 |
|------|---------|------|
| `src/path-to-tarkov-controller.ts` | 14 | 含 syncLocationBase、getOffraidPosition、updateOffraidPosition 等热路径 |
| `src/event-watcher.ts` | 5 | 含 getEndOfRaidPayload、handleRegularExtracts 等战后路径 |
| `src/stash-controller.ts` | 2 | 含 stash 切换调试日志 |
| `src/traders-controller.ts` | 1 | trader 更新调试日志 |

**不修改的例外**：
- 真正的 warning/error 日志保持原级别不变。
- 用户可见的操作日志（如 stash 切换成功、transit 完成等）保持 `info` 级别，但去掉 `[PTT Debug]` 前缀，改为 `[PTT]`。

**回退方式**：全局搜索 `logger.debug(\`\[PTT Debug\]` 恢复为 `logger.info`。

## 阶段 4：性能优化（延后）

### 4.1 MEDIUM-3：热路径 deepClone 开销

- **文件**：`src/path-to-tarkov-controller.ts`（第 106 行 `getConfig`、第 490 行 `createGenerateAll`、第 522-549 行 `createGetTemplateItems`）
- **风险**：MEDIUM
- **状态**：DEFER

**延后原因**（三议员一致同意）：

1. **当前性能影响可接受**：在典型配置（40 个 offraid positions、12 张地图）下，单次 `deepClone` 耗时约 3-8ms。考虑到每次玩家进战局只触发一次配置加载，对用户体验的影响可以忽略。
2. **部分克隆引入共享状态风险**：改为浅克隆或 Proxy 模式需要彻底重构 `getConfig()` 的返回值使用方式。当前代码中存在多处直接修改返回配置对象的模式（`config.exfiltrations[mapName] = ...`），如果改为共享 baseConfig + delta 模式，这些变异操作会污染 baseConfig，引入难以追踪的 bug。
3. **预期工作量大**：估计重构需要修改 15+ 个调用点，增加测试覆盖，且改动风险高。当前阶段的人力应集中在阶段 1-3 的用户可见问题上。

**标记为技术债**：在 `src/path-to-tarkov-controller.ts` 文件顶部添加 `@PERF-TODO` 注释，指向 MEDIUM-3 的追踪 issue。

**后续重构方向（备忘）**：
- `getConfig()`：baseConfig 冻结后共享，session 级配置通过 `Object.assign({}, baseConfig, sessionDeltas[sessionId])` 创建。
- `createGenerateAll()`：不克隆整个 `result.locations`，改为对每个 locationBase 创建轻量副本（`{ ...locationBase, exits: modifiedExits }`）。
- `createGetTemplateItems()`：缓存解析后的 `items` 对象，后续请求复用。在 SPT 发布更新时通过 `configServer` 的版本号触发缓存失效。
