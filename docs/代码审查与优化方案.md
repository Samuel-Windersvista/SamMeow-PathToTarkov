## 严重问题（CRITICAL）

### [CRITICAL-1] CurrentLocationDataService 失败后仍标记初始化成功

- **文件**: `PTT-Plugin/Services/CurrentLocationDataService.cs`（第 14-25 行）
- **严重性**: CRITICAL
- **分类**: 逻辑错误 / 数据一致性问题

**问题描述**:

`Init()` 方法的控制流存在致命缺陷：

```csharp
public void Init()
{
    if (_isInitialized) { /* skip */ return; }
    FetchExfilsTargetsForCurrentLocation();  // 内部 try/catch，吞掉异常
    _isInitialized = true;                   // 无论成功与否都标记为 true
    PTT.Patches.InitAllExfiltrationPointsPatch.TryApplyExfilFiltering();
}
```

`FetchExfilsTargetsForCurrentLocation()` 方法内部（第 78-87 行）有 try/catch，当 HTTP 请求失败时仅记录错误日志，不抛异常。调用方 `Init()` 无法获知 fetch 是否成功，继续将 `_isInitialized` 置为 `true`。

后果：当服务器首次启动时网络不通或后端未就绪，`CurrentLocationData` 保持为 `new CurrentLocationDataResponse()`（空的 `exfilsTargets` 字典）。`IsExfiltrationPointEnabled()` 对任何撤离点都返回 `false`，导致**所有撤离点被禁用**。玩家进入战局后无法撤离。

状态重置函数 `Reset()` 虽能将 `_isInitialized` 清零，但只在 `KeepFoundInRaidTweak` 相关路径被调用（通过 checkmark 补丁），没有自动重试机制。

**修复方案**:

1. 让 `FetchExfilsTargetsForCurrentLocation()` 返回 `bool` 类型，失败时返回 `false`。
2. 在 `Init()` 中检查返回值，只有成功时才设置 `_isInitialized = true` 并调用 `TryApplyExfilFiltering()`。
3. 增设指数退避重试逻辑，而非永久跳过。

```csharp
public bool Init()
{
    if (_isInitialized) return true;
    bool success = FetchExfilsTargetsForCurrentLocation();
    if (!success) {
        Logger.Error("CurrentLocationDataService init failed, exfil filtering skipped");
        return false;
    }
    _isInitialized = true;
    PTT.Patches.InitAllExfiltrationPointsPatch.TryApplyExfilFiltering();
    return true;
}
```

## 高危问题（HIGH）

### [HIGH-1] 战后流程双阶段 saveProfile 导致存档不一致

- **文件**: `src/keep-fir-tweak.ts` 第 17 行, `src/path-to-tarkov-controller.ts` 第 483 行
- **严重性**: HIGH
- **分类**: 数据一致性

**问题描述**:

玩家撤离时 `onPlayerExtracts()` 的执行路径：

```
onPlayerExtracts()
  -> KeepFoundInRaidTweak.setFoundInRaidOnEquipment()
       -> saveServer.saveProfile(sessionId)   // 第一次保存 (FIR 标记)
  -> updateOffraidPosition()
       -> stashController.updateStash()
       -> tradersController.updateTraders()
       -> saveServer.saveProfile(sessionId)   // 第二次保存 (stash/trader 更新)
```

如果第一次保存后、第二次保存前进程崩溃（服务器崩溃、网络中断等），存档处于半更新状态：物品的 FIR 标记已经写入但 stash 切换和 trader 状态尚未更新。玩家重新进入后会发现自己持有刚刚装备的 FIR 物品，但仓库位置和商人状态回到了战前状态，造成物品复制风险。

**根因分析**:

`KeepFoundInRaidTweak` 在构造函数中收到 `SaveServer` 引用后直接调用了 `saveProfile`，将「数据修改」和「持久化」耦合在一起。控制器层面也没有将这两个操作合并为一个原子操作。

**修复方案**:

1. 从 `KeepFoundInRaidTweak.setFoundInRaidOnEquipment()` 中移除 `saveServer.saveProfile()` 调用，让它只返回修改后的数据。
2. 将保存操作提升到控制器层面，统一在 `updateOffraidPosition()` 中一次性写入。

```typescript
// keep-fir-tweak.ts - 移除 save
public setFoundInRaidOnEquipment(sessionId: string, isPlayerScav: boolean): number {
  const profile = this.saveServer.getProfile(sessionId);
  // ... 修改物品 ...
  // 不在这里 save
  return nbImpactedItems;
}

// path-to-tarkov-controller.ts - 保持单一 save
onPlayerExtracts(params) {
  if (this.userConfig.gameplay.keepFoundInRaidTweak) {
    const firTweak = new KeepFoundInRaidTweak(this.saveServer);
    firTweak.setFoundInRaidOnEquipment(sessionId, isPlayerScav);
    // 不单独 save
  }
  this.updateOffraidPosition(sessionId, newOffraidPosition);
  // 只有 updateOffraidPosition 末尾调 saveProfile
}
```

### [HIGH-2] CurrentExfilTargetService 全局静态单例污染

- **文件**: `PTT-Plugin/Services/CurrentExfilTargetService.cs`
- **严重性**: HIGH
- **分类**: 架构缺陷

**问题描述**:

```csharp
public static class CurrentExfilTargetService
{
    private static ExfilTarget CurrentExfilTarget { get; set; } = null;
    // ...
}
```

这是一个进程级别的全局静态变量。单个 SPT 服务器进程可能服务多个玩家会话（在 cooperative 模式或 headless 客户端场景）。静态 `CurrentExfilTarget` 会在不同玩家的撤离请求之间共享：

- 玩家 A 调用 `SaveExfil(targetA)`，玩家 B 随后调用 `SaveExfil(targetB)`。
- 玩家 A 的撤离请求如果稍有延迟，`ConsumeExitName()` 可能返回玩家 B 的数据。
- `SaveExfil` 在确认 transit 控制器存在之前就保存了撤离点，失败路径没有任何清理逻辑。

**修复方案**:

1. 改为 Per-Session 存储模式：使用 `ConcurrentDictionary<string, ExfilTarget>` 以 `sessionId` 为键。
2. 只有确认 transit 控制器响应成功后才写入存储。
3. 在 `Reset()` 或 `Init()` 中提供按 sessionId 的显式清理方法。

### [HIGH-3] headless 客户端通过用户名匹配串位

- **文件**: `src/path-to-tarkov-controller.ts` 第 1073-1103 行
- **严重性**: HIGH
- **分类**: 逻辑缺陷 / 多玩家兼容性

**问题描述**:

```typescript
const profileName = profile?.info?.username || '';
const isHeadless = profileName.toLowerCase().includes('headless');

if (isHeadless) {
  const profiles = this.saveServer.getProfiles();
  for (const [otherSessionId, otherProfile] of Object.entries(profiles)) {
    const otherProfileName = otherProfile?.info?.username || '';
    if (!otherProfileName.toLowerCase().includes('headless')

```

依赖用户名包含 "headless" 子串来识别 headless 客户端有严重问题：
- 任何用户名包含 "headless" 的正常玩家会被误判。
- 遍历所有 profiles 找到第一个非 headless 且有 offraidPosition 的 profile，在多人环境下可能匹配到错误的玩家。

**修复方案**:

1. 在 SPT 服务端层面建立显式的 session/profile/group 映射关系（例如在 raid 创建时记录主玩家和 headless 的 session 配对）。
2. 将 headless 的 sessionId 与主玩家的 sessionId 在 RaidCache 中关联。
3. 消除对用户名模式的依赖。

### [HIGH-4] 配置校验 typo 导致漏检坏 transit target

- **文件**: `src/config-analysis.ts` 第 146 行
- **严重性**: HIGH
- **分类**: 配置校验

**问题描述**:

```typescript
if (!offraidPosition && (!parsed.transitTargetMapName || !parsed.transitTargetMapName)) {
  errors.push(`cannot parse exfil target in exfiltrations.${mapName}.${extractName}`);
}
```

条件中 `!parsed.transitTargetMapName` 被检查了两次，而 `transitTargetSpawnPointId` 完全没有被检查。这意味着：

- 如果用户配置了一个有 mapName 但缺少 spawnPointId 的 transit target（例如 `"bigmap."` 这种格式），校验不会报错。

```typescript
if (!offraidPosition && (!parsed.transitTargetMapName || !parsed.transitTargetSpawnPointId)) {
```

### [HIGH-5] end-of-raid transit/extract 判定逻辑缺陷

- **文件**: `src/event-watcher.ts` 第 279-304 行, `src/end-of-raid-controller.ts` 第 54-67 行
- **严重性**: HIGH
- **分类**: 逻辑缺陷

**问题描述**:

问题一：在 `event-watcher.ts` 的 `getEndOfRaidPayload()` 中（第 297 行）：

```typescript
const endOfRaidPayload: EndOfRaidPayload = {
  // ...
  newOffraidPosition: targetOffraidPosition,
  isTransit: !targetOffraidPosition,   // 只用 targetOffraidPosition 判定 transit
};
```

transit 的判定完全依赖于 `targetOffraidPosition` 是否为 null，而没有使用 `exitStatus === 'Transit'` 来辅助判定。在 vanilla transit 场景下，如果 exitName 不包含 transit 信息（没有使用 `.` 分隔的 PTT 格式），`targetOffraidPosition` 可能为 null，`isTransit` 可能被错误设为 `true`。

问题二：`handleRegularExtracts()`（第 225-251 行）被设计用于处理没有安装 Interactable Exfils API 时的回退逻辑，但它通过 `parseExilTargetFromPTTConfig` 在 exfil 配置中找到 `targetOffraidPosition` 非空的目标。然而在第 300 行的调用条件中：

```typescript
if (exitName && !targetOffraidPosition && !transitTargetMapName && !transitTargetSpawnPointId) {
  return this.handleRegularExtracts(endOfRaidPayload);
}
```

这个条件意味着「有 exitName，但没有 PTT 格式解析结果」，但这可能恰好是 vanilla transit 的出口（exit name 不是 PTT 格式但确实是一个有效撤离名）。vanilla transit 会被错误路由到 `handleRegularExtracts`。

问题三：在 `end-of-raid-controller.ts` 中（第 54 行），`newOffraidPosition` 非空时调用 `onPlayerExtracts` 触发 FIR tweak 和位置更新。但 `isTransit` 为 true 时只是日志记录，不执行任何动作——然而此时如果没有调用 `onPlayerExtracts`（当 `newOffraidPosition` 为 null 且 isTransit 为 false 时仅记 warning），返回的玩家将得不到位置更新。

**修复方案**:

1. 优先使用 `raidCache.exitStatus === 'Transit'` 来判定 transit，而非依赖于 `targetOffraidPosition`。
2. 在 `handleRegularExtracts` 入口处增加检查：如果 `exitStatus === 'Transit'`，跳过回退逻辑。
3. 在 `end-of-raid-controller.ts` 中对 transit 路径补充位置更新（至少确保 offraidPosition 不变）。

```typescript
// event-watcher.ts getEndOfRaidPayload
const isTransit = raidCache.exitStatus === 'Transit'
  || (!targetOffraidPosition && transitTargetMapName !== null);

// 修正调用条件
if (exitName && !targetOffraidPosition && !transitTargetMapName && !transitTargetSpawnPointId
    && raidCache.exitStatus !== 'Transit') {
  return this.handleRegularExtracts(endOfRaidPayload);
}
```

## 中危问题（MEDIUM）

### [MEDIUM-1] AccessVia 字符串匹配缺陷

- **文件**: `src/helpers.ts` 第 15-17 行
- **严重性**: MEDIUM
- **分类**: Bug

**问题描述**:

```typescript
export function checkAccessVia(access_via: AccessVia, value: string): boolean {
  return isWildcardAccessVia(access_via) || access_via.includes(value);
}
```

`AccessVia` 的类型是 `string | string[]`。当 `access_via` 是字符串时，JavaScript 的 `String.prototype.includes()` 执行的是**子串匹配**而非精确值匹配。例如：

- `access_via = "factory4_day"`, `value = "factory"` => `"factory4_day".includes("factory")` 返回 `true`。
- 如果配置中的 `offraid_regen_config.hydration.access_via` 被写为 `"MY_TOWN"`，而某个 offraidPosition 是 `"MY_TOWNHALL"`，那么 `"MY_TOWNHALL".includes("MY_TOWN")` 错误返回 `true`。

在多数标准配置中, `access_via` 经过 `fromRawExfiltrations` 和 `ensureArray` 处理后已经是数组格式，数组的 `includes()` 进行的是精确匹配，行为正确。但调用方可能直接传递字符串字面量（如某些内联的 `access_via` 配置没有经过 normalize 流程），导致非预期的 substring 匹配。

**修复方案**:

```typescript
export function checkAccessVia(access_via: AccessVia, value: string): boolean {
  const viaArray = Array.isArray(access_via) ? access_via : [access_via];
  return isWildcardAccessVia(access_via) || viaArray.includes(value);
}
```

此修复确保总是使用数组形式的精确包含匹配，消除类型差异带来的语义歧义。

### [MEDIUM-2] raidCaches / configCache 无清理导致内存泄漏

- **文件**: `src/event-watcher.ts` 第 41 行, `src/path-to-tarkov-controller.ts` 第 80 行
- **严重性**: MEDIUM
- **分类**: 资源泄漏

**问题描述**:

```typescript
// event-watcher.ts
private raidCaches: Record<string, RaidCache>;  // 永不收缩

// path-to-tarkov-controller.ts
private configCache: Record<string, Config> = {};  // 永不收缩
```

两个缓存都用 sessionId 索引，但没有任何移除机制。在 SPT 长期运行的服务器中，每次新玩家连接或创建 profile 都会向这两个字典添加新条目。如果服务器持续运行数小时并服务数百个不同的玩家会话，内存消耗将持续增长。

`raidCaches` 的问题尤其严重：每个 RaidCache 对象包含 `currentLocationName`、`exitStatus` 等字段，并且每个缓存的条目在玩家断开后不再有任何用处。如果玩家反复进出战局，每次都会通过 `createInitialRaidCache` 创建新条目（但 `initRaidCache` 中有一个特例：`exitStatus === 'Transit'` 时跳过 reset，但不会清理旧条目）。

**修复方案**:

1. 在 `EventWatcher` 中增加 session 生命周期钩子：当玩家断开连接或 profile 被卸载时，删除对应的 `raidCaches[sessionId]`。
2. `configCache` 可配合 SPT 的 `OnProfileDeleted` 或类似事件进行清理。
3. 缓存改用 `Map<string, T>` 以便更高效地迭代和删除。

```typescript
public cleanupSession(sessionId: string): void {
  delete this.raidCaches[sessionId];
  // 通知控制器清理 configCache
}
```

### [MEDIUM-3] 热路径上的 deepClone / JSON.parse 开销

- **文件**: `src/path-to-tarkov-controller.ts` 第 106、490、522-549 行
- **严重性**: MEDIUM
- **分类**: 性能

**问题描述**:

三个关键热路径存在不必要的全量深拷贝：

1. **`getConfig()`**（第 106 行）: 每次新 session 请求配置时，`deepClone(this.baseConfig)` 克隆整个配置树。`baseConfig` 包含所有地图的 `infiltrations`、`exfiltrations`、`traders_config` 等完整数据。如果配置较大（通常数千行），这会产生显著的 GC 压力。

2. **`createGenerateAll()`**（第 490 行）: `deepClone(result.locations)` 克隆所有地图的完整 location 数据。`generateAll` 是每次玩家打开地图选择界面的必经路径。

3. **`createGetTemplateItems()`**（第 522-524、549 行）: 接收到完整的 items JSON 后 `JSON.parse(rawResult)` 解析，修改后 `JSON.stringify(parsed)` 重新序列化。这个调用链在玩家每次打开仓库或 stash 时触发，涉及数万条 item template 的序列化/反序列化。

**修复方案**:

1. **`getConfig()`** 改为「base config + session delta」模式：baseConfig 保持不可变引用，只对需要按 session 定制的字段进行浅拷贝或代理。

2. **`createGenerateAll()`** 改为按需克隆：只对 `locations` 中每个 `locationBase` 的少量字段（`Locked`、`Enabled`、`SpawnPointParams`、`exits`、`transits`）进行修改，使用 `Object.assign` 创建新的 locationBase 对象而非深度克隆整个 `result.locations`。

3. **`createGetTemplateItems()`** 改为懒解析模式：不解析完整 JSON，使用流式或按需查找的方式修改 stash size。如果必须解析，则应缓存解析结果并在后续请求中复用。

```typescript
// 优化 getConfig 示例
getConfig(sessionId) {
  const base = this.baseConfig;
  const delta = this.configDeltas[sessionId];
  if (!delta) {
    this.configDeltas[sessionId] = {};
    return base; // 首次直接返回 base（不可变）
  }
  // 合并 base + delta（浅合并）
  return { ...base, ...delta };
}
```

### [MEDIUM-4] 无匹配仓库时回退到 EMPTY_STASH

- **文件**: `src/stash-controller.ts` 第 117-126 行
- **严重性**: MEDIUM
- **分类**: 可用性

**问题描述**:

```typescript
private getSecondaryStash(offraidPosition: string, sessionId: string): Omit<StashConfig, 'access_via'> {
  return this.getConfig(sessionId).hideout_secondary_stashes.find(
    stash => checkAccessVia(stash.access_via, offraidPosition),
```

当 `hideout_secondary_stashes` 中没有匹配当前 `offraidPosition` 的配置时，方法静默返回 `EMPTY_STASH`（size=0 的特殊快）。这意味着：

- 如果 `multiStash` 启用且主 stash 在该位置不可用，玩家会获得一个容量为 0 的仓库。
- 玩家看到的仓库界面渲染正常但无法放入任何物品，表现为「隐形仓库」。
- 如果 SPT 的库存系统尝试将默认物品放入 size=0 的 stash，可能触发索引越界或 Item placement 异常。

**修复方案**:

1. 在返回 `EMPTY_STASH` 前输出 warning 日志，指明当前 `offraidPosition` 没有匹配的 secondary stash。
2. 考虑增加 `ROAMING_EMERGENCY_STASH` 或设备用回退仓库（如大小为 5 的小仓库）。
3. 在配置文件校验阶段（`analyzeConfig`）检查所有 `offraid_positions` 是否都有对应的 stash 覆盖。

### [MEDIUM-5] createGetTemplateItems 模板缺失导致硬崩溃

- **文件**: `src/path-to-tarkov-controller.ts` 第 536-547 行
- **严重性**: MEDIUM

**问题描述**:

```typescript
});
```

假定 `VANILLA_STASH_IDS` 中所有 ID 在数据库的 `templates.items` 中都存在且包含 `Grids[0]._props`。如果 SPT 升级后移除了某个 stash template，或修改了 items 的数据结构，这个循环会在 `gridProps` 为 falsy 时抛出一个未捕获异常。

这个异常发生在 `createGetTemplateItems` 中，它在 `DataCallbacks.getTemplateItems` 的 override 路径中被调用。任何玩家打开仓库都会触发这个路由，一旦抛出异常，该玩家的所有 HTTP 请求都会得到 500 错误。

**修复方案**:

```typescript
VANILLA_STASH_IDS.forEach(stashId => {
  const item = items[stashId];
  if (!item) {
    this.debug(`stash template ${stashId} not found, skipping`);
    return;
  }
  const grid = item?._props?.Grids?.[0];
  const gridProps = grid?._props;
  if (gridProps) {
    gridProps.cellsV = size;
  } else {
    this.debug(`cannot set size for stash template ${stashId}, missing grid props`);
  }
});
```

优雅降级而非硬崩溃，同时保留日志以便排查。

### [MEDIUM-6] 构造函数内部重复 new 依赖

- **文件**: `src/path-to-tarkov-controller.ts` 第 112-127 行
- **严重性**: MEDIUM
- **分类**: 架构

**问题描述**:

构造函数接收 `tradersAvailabilityService` 作为参数（第 89 行），但在第 112 行立即用 `new TradersAvailabilityService()` 覆盖了它：

```typescript
constructor(
  // ...
  private tradersAvailabilityService: TradersAvailabilityService,  // 注入的参数
  // ...
) {
  // ...
  this.tradersAvailabilityService = new TradersAvailabilityService();  // 覆盖！！
  this.stashController = new StashController(this.getConfig, userConfig, db, saveServer, this.debug);
  this.tradersController = new TradersController(this.tradersAvailabilityService, userConfig, db, saveServer, configServer, this.logger);
}
```

同样，`StashController` 和 `TradersController` 直接在构造函数中 `new` 创建，而不是通过 DI 容器注入。这使得：

- 单元测试无法 mock 这些依赖。
- 外部调用者传入的 `tradersAvailabilityService` 被无视，造成迷惑。
- `TradersAvailabilityService` 被创建两次（一次注入一次内部 new），浪费内存。

**修复方案**:

1. 使用注入的 `tradersAvailabilityService` 而非覆盖。
2. 将 `StashController` 和 `TradersController` 的创建迁移到 `mod.ts` 或工厂函数中，通过 DI 容器注册。

### [MEDIUM-7] 过多 PTT Debug 日志在 info 级别污染

- **文件**: 多个文件（`path-to-tarkov-controller.ts`、`event-watcher.ts` 等）
- **严重性**: LOW（MEDIUM in aggregate）
- **分类**: 可维护性

**问题描述**:

代码库中存在大量 `[PTT Debug]` 前缀的日志，统一使用 `logger.info()` 级别输出：

```typescript
this.logger.info(`[PTT Debug] syncLocationBase called for sessionId: ${sessionId}`);
this.logger.info(`[PTT Debug] RaidCache exists: ${!!raidCache}, exitStatus: ${raidCache?.exitStatus}, ...`);
this.logger.info(`[PTT Debug] Configured spawn points for ${mapName}: ${...}`);
```

在正常运行的服务器中，每次 raid 开始都会输出 10-20 条 `[PTT Debug]` 日志，包括同步前后两次打印 `SpawnPointParams` 长度和所有 player spawn 点的完整列表（第 131-137 行）。这些日志：

- 在 `info` 级别输出，无法被服务器管理员常规过滤（除非修改全局日志级别）。
- 日志字符串即使在不输出时（因日志级别过滤）也会被构造，产生不必要的字符串拼接开销。
- 对最终用户没有诊断价值，徒增日志文件大小。

**修复方案**:

1. 将 `[PTT Debug]` 日志统一降级到 `debug` 级别，或使用 mod 专门的调试开关。
2. 使用懒求值的日志模式：`this.logger.debug(() => `[PTT Debug] ...`)` 避免在禁用日志时构造字符串。
3. 对高频路径（如 `syncLocationBase` 中的详细 spawn 点日志）增加采样或仅在首次/出错时输出。

## 性能优化建议

### 1. 撤离点数据启动时缓存

当前所有撤离点数据通过 JSON 文件读入，每次获取时使用 `fs.existsSync` + `fs.readFileSync`（`exfils-targets.ts` 第 222-233 行 `getAllExtractsFromExternalResources`）。

```typescript
const getAllExtractsFromExternalResources = (mapName: string): string[] => {
  const externalResourcesPath = path.join(__dirname, '..', 'external-resources', 'maps', `${mapName}_allExtracts.json`);
  if (fs.existsSync(externalResourcesPath)) {
    const extractsData = JSON.parse(fs.readFileSync(externalResourcesPath, 'utf8'));
    // ...
  }
};
```

这是一个同步文件 I/O 操作，在 `getExfilsTargets` 的每请求路径中被调用。建议在启动时一次性读取并索引所有地图的撤离点数据，存入一个 `Map<string, string[]>` 中重复使用。

### 2. configCache 采用 base + session delta 模式

当前 `getConfig()` 每次都 `deepClone` 整个 `baseConfig`（第 106 行）。对于拥有 40 个 offraid positions 和 12 张地图的大型配置，单次 deepClone 耗时可达 5-15ms（取决于对象深度）。如果 10 个玩家同时进入战局，控制器会被串行阻塞 50-150ms。

改为「base config + session delta」模式：

```
baseConfig (immutable, shared reference)
  + sessionDeltas[sessionId] (只存储变化的部分)
  = effective config (通过 Proxy 或 Object.assign 动态合并)
```

### 3. 减少热路径上的 deepClone 范围

`createGenerateAll()` 中 `deepClone(result.locations)` 是最高开销的热点之一。建议：

- 不克隆整个 locations 对象，而是对每个 `locationBase` 创建一个「视图包装」。
- 对于只需要修改 `Locked`、`Enabled`、`exits` 等顶层字段的场景，使用 `{ ...locationBase, exits: modifiedExits }` 创建轻量副本。
- 将 `syncLocationBase` 的副作用（修改 spawn points、exits、transits）延迟到实际使用时应用。

### 4. 批量存档操作合并为单次保存

当前 `onPlayerExtracts` 路径在 FIR tweak 之后立即调用 `saveProfile`，然后 `updateOffraidPosition` 再次调用 `saveProfile`（见 HIGH-1）。除此之外，`StashController.updateStash` 和 `TradersController.updateTraders` 操作自身也可能触发多次本地持久化。

优化方案：

- 引入一个「事务批处理」API：`batchSave.begin()` → 执行所有修改 → `batchSave.commit()` 一次性写入。
- 或者使用 `SaveServer.saveProfile` 的延迟写（debounce）机制，在同一个 event loop tick 内的多次调用合并为一次实际 I/O。

## 已实施方案：ROAMING_EMERGENCY_STASH 应急仓（对应 MEDIUM-4）

### 问题背景

当玩家处于既无主仓权限、又无专属副仓定义的位置（如 `woods_FactGate_factory_Mtent`）时，PTT 的 `StashController.getSecondaryStash()` 会兜底到 `EMPTY_STASH`（size=0 的空仓）。这导致 SPT 核心在 `/client/game/profile/list` 处理 RagFair 到期回收/保险返还/任务物品投放时，因目标容器无可用空间而崩溃。

> 错误信息：`Could not find free space in grid hideout in item Standard stash 10x30 for item patron_545x39_BP`

### 设计决策

引入 `ROAMING_EMERGENCY_STASH`（固定 size=20 应急仓）作为 fallback 方案，取代 `EMPTY_STASH` 空仓。应急仓定位为**防崩溃缓冲**，非长期存储仓：

- 仅在无主仓权限且无匹配副仓时启用
- 离开该位置时自动递归清空内容，保留 root item
- 不自动搬运到主仓/副仓
- size=20 确保 RagFair/保险/任务返还至少有容器可落地

### 代码改动（6 个文件，+336 / -8 行）

**1. `src/config.ts` — 新增常量**

```ts
export const ROAMING_EMERGENCY_STASH = toStashConfig({
  id: 'PathToTarkov_Roaming_Emergency_Stash',
  size: 20,
  access_via: [],
});
```

**2. `src/stash-controller.ts` — 核心逻辑**

- `initSecondaryStashTemplates()` 注册应急仓模板：`[EMPTY_STASH, ROAMING_EMERGENCY_STASH, ...givenStashConfigs]`
- `initProfile()` 包含应急仓到内置 stash 列表
- `getSecondaryStash()` fallback 从 `EMPTY_STASH` 改为 `ROAMING_EMERGENCY_STASH`
- `updateStash()` 传递 `[ROAMING_EMERGENCY_STASH, ...secondaryStashes]` 给 `setInventorySlotIds()`
- 新增方法：
  - `shouldUseRoamingEmergencyStash()` — 判断某位置是否会兜底到应急仓
  - `isRoamingEmergencyStashActive()` — 当前活跃仓是否为应急仓
  - `clearRoamingEmergencyStashOnExit()` — 递归清空应急仓内容，保留 root

**3. `src/path-to-tarkov-controller.ts` — 接入脱战切换**


**5. `src/uninstall.ts` — 卸载时包含应急仓**
